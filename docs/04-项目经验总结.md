## 一、项目功能

### 已实现功能

| 功能 | 说明 |
|---|---|
| 文件上传 | 选择文件 → 上传到服务器 → MongoDB 记录文件信息 |
| 文件列表 | 从 MongoDB 查询，Redis 缓存 60 秒加速 |

| 健康检查 | `/api/health` 接口检测 MongoDB 和 Redis 状态 |

| 方法 | 路径 | 功能 |
|---|---|---|
| POST | `/api/upload` | 上传文件 |
| GET | `/api/files` | 获取文件列表 |
| GET | `/api/download/:id` | 下载文件 |
| DELETE | `/api/files/:filename` | 删除文件 |
| GET | `/api/health` | 健康检查 |

## 二、架构设计

### 整体架构

```
用户浏览器
    │
    │  http://8.162.9.143
    ▼
┌─────────────────────────────────────┐
│  Nginx（前端容器，端口 80）           │
│  ├── 静态文件：Vue 打包的 HTML/JS/CSS │
│  └── 反向代理：/api/* → 后端:8080     │
└─────────────┬───────────────────────┘
              │
              ▼
┌─────────────────────────────────────┐
│  Go + Gin（后端容器，端口 8080）      │
│  ├── 处理 API 请求                   │
│  ├── 文件存储到 /app/uploads         │
│  ├── 记录存入 MongoDB                │
│  └── 缓存存入 Redis                  │
└──────┬──────────────┬───────────────┘
       │              │
       ▼              ▼
┌────────────┐  ┌───────────┐
│  MongoDB   │  │   Redis   │
│  存文件记录 │  │  缓存加速  │
│  端口 27017 │  │  端口 6379 │
└────────────┘  └───────────┘
```

### 文件上传/下载流程

```
上传流程：
用户选择文件 → 前端 POST /api/upload → 后端生成随机文件名存磁盘
→ 原始文件名 + 随机文件名(StoreName) 存入 MongoDB → 清除 Redis 缓存 → 返回成功

下载流程：
用户点击下载 → 前端请求 GET /api/download/:id → 后端查 MongoDB 找到 StoreName
→ 读取磁盘文件 → 设置 Content-Disposition 为原始文件名 → 返回文件流

删除流程：
用户点击删除 → 前端 DELETE /api/files/:filename → 后端查 MongoDB 找到 StoreName
→ 删除磁盘文件 → 删除 MongoDB 记录 → 清除 Redis 缓存 → 返回成功
```

### 为什么需要 StoreName？

```
问题：用户上传两个同名文件（比如两个 "报告.pdf"），后面的会覆盖前面的
解决：给每个文件生成唯一的磁盘名（如 1773299953320592254.pdf）
关系：MongoDB 中同时保存原始文件名和磁盘文件名

FileRecord {
    Filename:  "报告.pdf"                    ← 原始名，给用户看
    StoreName: "1773299953320592254.pdf"     ← 磁盘名，实际存储用
}
```

## 三、CI/CD 自动化部署

### GitHub Actions 工作流程

```
开发者 git push 到 main 分支
         │
         ▼
GitHub Actions 触发 (deploy.yml)
         │
    ┌────┴─────┐
    │ Step 1   │  拉取代码 (checkout)
    │ Step 2   │  登录 GitHub Container Registry (ghcr.io)
    │ Step 3   │  构建后端 Docker 镜像并推送到 ghcr.io
    │ Step 4   │  构建前端 Docker 镜像并推送到 ghcr.io
    │ Step 5   │  SSH 到阿里云服务器：
    │          │    → 登录 ghcr.io
    │          │    → 拉取最新镜像
    │          │    → 停掉旧容器
    │          │    → 启动新容器
    │          │    → 清理旧镜像
    └──────────┘
```

### 关键配置

**GitHub Secrets（在仓库 Settings → Secrets 中配置）：**

| Secret 名称 | 说明 |
|---|---|
| `SERVER_HOST` | 服务器 IP 地址 |
| `SERVER_SSH_KEY` | 服务器 SSH 私钥（.pem 文件内容） |
| `GHCR_TOKEN` | GitHub Personal Access Token（需要 Packages 读写权限） |

**镜像地址：**

| 镜像 | 地址 |
|---|---|
| 后端 | `ghcr.io/m13629812143/image-group/backend:latest` |
| 前端 | `ghcr.io/m13629812143/image-group/frontend:latest` |

### 为什么用 ghcr.io 而不是 Docker Hub？

```
Docker Hub → 中国服务器无法访问（被墙）
阿里云镜像服务 → 需要额外注册配置
ghcr.io → 阿里云服务器可以访问 + GitHub Actions 原生支持（用 GITHUB_TOKEN 直接登录）
```

### 本地 vs 服务器 compose 文件的区别

| 对比 | docker-compose.yml（本地） | docker-compose.prod.yml（服务器） |
|---|---|---|
| 前后端镜像 | `build: ./backend` 本地构建 | `image: ghcr.io/...` 从仓库拉取 |
| 前端端口 | `3000:80`（本地避免端口冲突） | `80:80`（服务器直接用 80） |

## 四、踩坑记录与解决方案

### 坑 1：Windows WSL 版本过旧，Podman 启动失败

```
报错：WSL 2 requires an update to its kernel component
原因：Windows 的 WSL2 内核版本太老
解决：管理员 PowerShell 运行 wsl --update --web-download
```

### 坑 2：中国服务器无法访问 Docker Hub

```
报错：podman pull 超时
原因：Docker Hub 在国内被封锁
解决方案一：本地 podman save 导出 → scp 传到服务器 → podman load 导入
解决方案二：使用 ghcr.io（GitHub Container Registry），中国可以访问
最终采用：方案二（配合 GitHub Actions 自动推送）
```

### 坑 3：服务器 yum install docker 装的是 Podman

```
现象：运行 yum install docker，实际装了 podman-docker
原因：Alibaba Cloud Linux 3 用 Podman 替代了 Docker
影响：无影响，Podman 完全兼容 Docker 命令
注意：compose 命令用 podman-compose 而不是 docker compose
```

### 坑 4：Podman 无法绑定 80 端口

```
报错：rootless 模式不允许绑定 1024 以下端口
原因：Windows 上 Podman 以普通用户运行
解决：本地开发用 3000 端口映射（3000:80），服务器上 root 用户直接用 80
```

### 坑 5：Go 版本与 Docker 镜像不匹配

```
报错：go.mod requires go >= 1.26
原因：Dockerfile 中写了 golang:1.23-alpine，版本太低
解决：改用 golang:1-alpine（自动拉取最新版本）
```

### 坑 6：Vue/Vite 需要 Node.js 20+

```
报错：Vite requires Node.js 20 or higher
原因：Dockerfile 中用了旧版 node 镜像
解决：改用 node:22-alpine
```

### 坑 7：SSH 登录服务器用户名不对

```
报错：Permission denied (publickey)
原因：阿里云 ECS 默认用户是 root（不是 admin 或 ec2-user）
解决：ssh -i 密钥.pem root@服务器IP
```

### 坑 8：浏览器缓存导致页面不更新

```
现象：服务器已部署新版本，但浏览器显示旧页面
原因：浏览器缓存了旧的 HTML/JS/CSS
解决：Ctrl + Shift + R 强制刷新（或 Ctrl + F5）
```

### 坑 9：删除文件只删了数据库记录，没删磁盘文件

```
问题：deleteFile() 只执行了 MongoDB DeleteOne，磁盘上的文件还在
后果：uploads 目录越来越大，磁盘空间浪费
解决：删除前先查 MongoDB 拿到 StoreName，再 os.Remove 删除磁盘文件
```

### 坑 10：GitHub Actions 创建 PAT 时缺少 Packages 权限

```
问题：Fine-grained token 中找不到 Packages 权限选项
原因：Fine-grained token 对 Packages 支持有限
解决：虽然跳过了该选项，但 token 仍然可用（仓库是 public 的）
备选：使用 Classic token 并勾选 write:packages 权限
```

## 八、学到的核心经验

### 1. 容器化是部署的关键

不用容器的话，你需要在服务器上装 Go、Node.js、MongoDB、Redis、Nginx... 每个的版本还要对得上。
用了容器之后，一个 `podman-compose up -d` 就全部搞定，环境完全一致。

### 2. CI/CD 节省大量时间

没有 CI/CD 之前，每次更新要：
```
本地构建 → podman save 导出镜像 → scp 传到服务器（几百 MB）→ 服务器 podman load → 重启
```

有了 GitHub Actions 之后：
```
git push → 自动完成上面所有步骤（约 7 分钟）
```

### 3. 缓存是性能优化的第一步

Redis 缓存文件列表 60 秒，在频繁访问的场景下可以减少 MongoDB 查询几十倍。
响应头 `X-Cache: HIT` 表示命中缓存（约 2ms），`MISS` 表示查了数据库（约 200ms）。

### 4. 前后端分离的通信方式

```
前端（Vue）和后端（Gin）完全独立运行
通信方式：前端用 axios 发 HTTP 请求给后端 API
开发时：前端直连 http://localhost:8080/api（跨域，需要 CORS）
生产时：Nginx 反向代理 /api/ → backend:8080（同域，无跨域问题）
```

### 5. 开发环境和生产环境要分开配置

| 配置 | 开发环境 | 生产环境 |
|---|---|---|
| compose 文件 | docker-compose.yml | docker-compose.prod.yml |
| 前端端口 | 3000 | 80 |
| 镜像来源 | 本地 build | ghcr.io 拉取 |
| API 地址 | http://localhost:8080/api | /api (Nginx 代理) |
| 数据库地址 | localhost:27017 | mongo:27017 (容器网络) |

### 6. 安全意识

- SSH 密钥登录比密码登录更安全
- 敏感信息（密钥、Token）放在 GitHub Secrets 中，不要写在代码里
- 生产环境不要暴露数据库端口到公网（当前 27017 和 6379 暴露了，建议后续关闭）
